2 ЦЕЛИ И НАЗНАЧЕНИЕ СОЗДАНИЯ АВТОМАТИЗИРОВАННОЙ СИСТЕМЫ

2.1 Цели создания АС

2.1.1 Технические показатели объекта автоматизации, которые должны быть достигнуты:

1. Показатель 1: Время составления учебного расписания
   · Текущее значение: 120 часов (15 рабочих дней)
   · Требуемое значение: 2 часа
   · Критерий достижения: Сокращение на 98,3%
2. Показатель 2: Точность расписания (отсутствие конфликтов)
   · Текущее значение: 5% случаев конфликтов групп
   · Требуемое значение: 0% случаев
   · Критерий достижения: Полная ликвидация конфликтов
3. Показатель 3: Время учета преподавательской нагрузки
   · Текущее значение: 84 часа в месяц (10,5 рабочих дней)
   · Требуемое значение: 43 минуты в месяц
   · Критерий достижения: Сокращение на 99,1%
4. Показатель 4: Точность расчета заработной платы
   · Текущее значение: 95%
   · Требуемое значение: 100%
   · Критерий достижения: Полная автоматизация расчетов
5. Показатель 5: Время внесения изменений в расписание
   · Текущее значение: 1-4 дня на одно изменение
   · Требуемое значение: 4-5 минут на одно изменение
   · Критерий достижения: Сокращение на 99,6%

2.1.2 Технологические показатели:

1. Автоматизация процесса составления расписания: с 10% до 95%
2. Охват пользователей системой: с 5 человек до 120+ человек
3. Доступность информации: с бумажных носителей до онлайн-доступа 24/7
4. Интеграция процессов: с изолированных процессов до единой системы

2.1.3 Производственно-экономические показатели:

1. Экономия трудозатрат: 200+ человеко-часов в год
2. Сокращение расходов на бумагу и канцелярию: 50 000 рублей в год
3. Повышение производительности труда администрации: на 30%
4. Снижение количества ошибок в расчетах: на 100%

2.1.4 Критерии оценки достижения целей создания АС:

1. Количественные критерии:
   · Время генерации расписания не превышает 10 минут
   · Отсутствие зафиксированных конфликтов в течение семестра
   · Удовлетворенность пользователей ≥90%
   · Количество автоматизированных процессов ≥95%
2. Качественные критерии:
   · Отсутствие жалоб на расписание от преподавателей и студентов
   · Полное соответствие расписания учебным планам
   · Своевременность формирования отчетности
   · Удобство использования системы

4 ТРЕБОВАНИЯ К АВТОМАТИЗИРОВАННОЙ СИСТЕМЕ

4.1 Требования к структуре АС в целом

4.1.1 Перечень подсистем (при их наличии), их назначение и основные характеристики

1. Подсистема управления данными (ПУД)
   · Назначение: Хранение, обработка и управление исходными данными
   · Основные характеристики:
     · Объем хранимых данных: до 10 ГБ
     · Количество записей: 100 000+
     · Время доступа к данным: ≤ 1 секунды
     · Поддержка транзакций: ACID-совместимость
2. Подсистема генерации расписания (ПГР)
   · Назначение: Автоматическое составление оптимального расписания
   · Основные характеристики:
     · Время генерации: ≤ 10 минут
     · Алгоритм оптимизации: генетический алгоритм
     · Количество итераций: до 1000
     · Точность: 100% отсутствие конфликтов
3. Подсистема учета нагрузки (ПУН)
   · Назначение: Учет и расчет преподавательской нагрузки
   · Основные характеристики:
     · Точность расчетов: 100%
     · Интеграция с 1С: автоматическая
     · Формирование отчетов: ≤ 5 минут
     · Поддержка различных ставок
4. Подсистема публикации и доступа (ППД)
   · Назначение: Обеспечение доступа к расписанию
   · Основные характеристики:
     · Количество одновременных пользователей: ≥ 100
     · Время отклика: ≤ 2 секунды
     · Форматы вывода: Web, PDF, Excel
     · Мобильная адаптация: есть

Требования к числу уровней иерархии и степени централизации АС:

· Количество уровней иерархии: 3 уровня
  1. Уровень представления (клиентский)
  2. Уровень бизнес-логики (серверный)
  3. Уровень данных (база данных)
· Степень централизации: централизованная система
· Архитектура: клиент-серверная
· Распределенность: центральный сервер с тонкими клиентами

4.1.2 Требования к способам и средствам обеспечения информационного взаимодействия компонентов АС

1. Протоколы взаимодействия:
   · HTTP/HTTPS для веб-доступа
   · TCP/IP для внутреннего взаимодействия
   · SMTP/POP3 для почтовых уведомлений
2. Форматы данных:
   · JSON для API взаимодействия
   · XML для интеграции с внешними системами
   · CSV для импорта/экспорта данных
3. Средства обеспечения:
   · API Gateway для управления запросами
   · Message Queue для асинхронной обработки
   · Load Balancer для распределения нагрузки
4. Требования к скорости обмена:
   · Локальная сеть: ≥ 100 Мбит/с
   · Внешний доступ: ≥ 10 Мбит/с
   · Задержка: ≤ 50 мс
